# 4. 운영 프로세스 및 정책 (How it works?)

## 4.1. 서비스 성격 및 책임 한계 (Service Definition)

본 서비스는 **"전문가 가용성 확보"**를 핵심 가치로 하는 리테이너(Retainer) 모델입니다. 고객사가 구매하는 것은 단순한 '작업 결과물'이 아니라, **'검증된 최고 등급 전문가를 언제든 활용할 수 있는 권리'**입니다.

### 1) SLA가 아닌 SLO 기반 운영 - 왜 이것이 더 합리적인가?

#### 기존 SLA 방식의 한계

전통적인 SLA(Service Level Agreement)는 **"시스템 가동률 99.9% 보장"**과 같이 결과를 법적으로 보장하고, 미달 시 페널티를 부과하는 구조입니다. 하지만 이는 다음과 같은 문제가 있습니다:

* **책임 소재 불명확**: IT 장애는 다양한 요인(네트워크, 하드웨어, 애플리케이션, 외부 API 등)이 복합적으로 작용하므로, 단일 벤더에게 전체 가동률 책임을 묻는 것은 비현실적
* **방어적 운영 유발**: 벤더는 페널티 회피를 위해 보수적으로 대응하게 되어, 오히려 **근본적 문제 해결보다 임시 조치**에 집중
* **높은 비용**: SLA 보장을 위한 리스크 프리미엄이 계약 금액에 반영되어 **불필요한 비용 증가**

#### 본 서비스의 SLO (Service Level Objective) 접근법

본 서비스는 **"대응 적시성(Response Time)"**에 집중합니다:

* **명확한 약속**: 장애 접수 후 **2시간 내** 최고 등급 엔지니어가 분석에 착수
* **측정 가능**: 대응 시작 시간은 명확히 기록되고 검증 가능
* **실질적 가치**: 고객이 진정으로 필요한 것은 "페널티"가 아니라 **"빠르고 정확한 문제 해결"**

> **고객 입장에서의 이점**: 페널티를 받는 것보다, 장애가 발생했을 때 **즉시 최고의 전문가가 투입되어 신속히 해결**되는 것이 훨씬 더 가치있습니다.

### 2) Advisory Role - 왜 직접 운영하지 않는가?

#### 역할 분담의 명확화

* **고객사의 권한과 책임**: 시스템에 대한 최종 통제권(명령어 실행, 설정 변경 등)은 고객사가 보유
* **당사의 역할**: 로그 분석, 원인 진단, 해결 방안 제시 등 **Tier 3 Expert Support**

#### 이러한 구조가 더 안전한 이유

* **보안 및 컴플라이언스**: 외부 벤더에게 프로덕션 시스템의 직접 접근 권한을 부여하는 것은 보안 정책상 위험
* **책임 소재 명확화**: 시스템 변경에 대한 최종 의사결정과 실행은 고객사가 하므로, 법적 책임 소재가 명확
* **지식 이전**: 고객사 담당자가 직접 조치를 수행하면서 **기술 노하우를 자연스럽게 습득**

> **실제 운영 방식**: 당사 전문가가 "이 쿼리를 실행하면 병목이 해소됩니다"라고 정확한 액션 아이템을 제시하면, 고객사 담당자가 검토 후 실행. 필요시 화면 공유 또는 현장 지원을 통해 실시간 가이드 제공.

### 3) Retainer Framework - 보험처럼 활용하는 전문가 그룹

#### 건당 과금 vs. 리테이너 계약 비교

| 구분 | 건당 과금 (Time & Material) | 리테이너 계약 (본 서비스) |
| ------ | --------------------------- | ------------------------ |
| **비용 예측성** | 장애 발생 시마다 별도 견적 → 예산 불확실 | 연간 고정 비용으로 **예산 예측 가능** |
| **대응 속도** | 계약 협의 → 인력 배정 → 작업 시작 (수일 소요) | 즉시 투입 (**2시간 내 착수**) |
| **전문가 품질** | 가용한 인력 배정 (품질 편차 큼) | **사전 검증된 최고 등급 전문가 전담** |
| **활용 범위** | 장애 대응에만 국한 | 정기 진단 + 장애 대응 **통합 지원** |

#### 핵심 가치: "안심(Peace of Mind)"

* 장애가 발생하지 않더라도, **정기 진단을 통해 잠재 리스크를 선제적으로 제거**
* 만약 장애가 발생하면, 별도 협의 없이 **즉시 최고의 전문가가 투입**
* 이는 마치 **"기술 보험"**과 같은 개념으로, 비용 대비 안정성 확보 효과가 탁월

## 4.2. 운영 프로세스 상세

### 정기 진단 서비스 프로세스

1. **진단 계획 수립** (진단 1개월 전)
    * 고객사와 진단 범위, 일정, 접근 권한 등 협의
2. **사전 자료 수집** (진단 2주 전)
    * 시스템 구성도, 최근 장애 이력, 성능 모니터링 데이터 등 수집
3. **현장 진단 수행** (1~2주)
    * 아키텍처 리뷰, 성능 분석, 보안 점검 등 실시
4. **결과 보고 및 개선안 제시** (진단 완료 후 1주 내)
    * 상세 진단 리포트 및 우선순위별 개선 권고안 제공
5. **후속 조치 지원** (필요시)
    * 개선 작업 수행 시 기술 자문 제공

### 장애 대응 서비스 프로세스

1. **긴급 요청 접수** (24/7 Hot-line)
    * 전용 연락처를 통해 장애 상황 접수
2. **전문가 투입** (접수 후 2시간 내)
    * 장애 유형에 맞는 최고 등급 엔지니어 즉시 배정
3. **원인 분석 및 조치 가이드** (실시간)
    * 로그 분석, 원인 파악, 해결 방안 제시
4. **복구 완료 확인**
    * 시스템 정상화 확인 및 재발 방지 대책 수립
5. **사후 분석 리포트** (복구 후 3일 내)
    * 근본 원인 분석 및 장기적 개선 방안 제시

### M/M 차감 및 관리 기준

* **진단 서비스**: 실제 투입된 공수(M/M)만큼 차감
* **장애 대응**: 최소 0.5 M/M 단위로 차감 (2시간 미만 소요 시에도 0.5 M/M 차감)
* **월별 리포트**: 매월 M/M 사용 현황 및 잔여 M/M 공유
* **이월 정책**: 연간 계약 기간 내 미사용 M/M은 차년도로 이월 가능 (최대 20%)
